Why this work existed

Why

The request was straightforward, but it needed supporting work

Natural-language questions

Staff want to ask plain-English questions about classes, students, subjects, teachers, location, attendance, and EL progression.

Local-first constraint

The solution needs to work with a local LLM, not a cloud-hosted assistant, so student data can stay on-site.

Proof before pilot

Before anyone trusts it with real data, we need evidence that the model can reliably answer the exact question types the school cares about.

In plain terms: the real question is not whether an LLM can discuss CSV files, but whether a local LLM can answer this school’s question set with acceptable reliability while keeping the data local.

Tooling

Tools

The local model relies on constrained local tools

The local LLM runs through LM Studio, but the answer quality depends on tool support: tightly-scoped CSV operations plus higher-level joined tools for the harder school questions.

Generic CSV tools
  • list_files and read_headers for schema discovery
  • query, summarise, distinct, and cross_tabulate for structured retrieval
  • Useful for straightforward class counts, breakdowns, and filters
Joined specialist tools
  • get_student_overview
  • get_student_teachers
  • get_student_location
  • find_students_same_subjects
  • get_level_teachers
Why that matters: the local model was less reliable when it had to work out multi-step joins by itself. Turning those joins into dedicated tools improved consistency.

Validation approach

Validation

The benchmark tested the full local setup, not just the raw model

14 benchmark questions

The same school-style questions were scored repeatedly against known ground truth computed directly from the synthetic CSVs.

Fact-based scoring

Answers were checked for the actual facts they contained, not for matching exact phrasing or markdown layout.

Safety rails

We added loop detection, tool-call limits, and one-shot answer forcing so the local model could not spiral into repeated tool use.

Important nuance: the benchmark does not claim that all local LLMs will perform well on this task. It shows that this specific local setup can answer this specific school question set well enough to justify the next step.

What improved the result

  • Specialist joined tools for the hard multi-file questions
  • Pre-seeded schema so the model wastes fewer calls on discovery
  • One-shot answer flow after high-confidence tools return
  • Cleaner scoring that recognises correct answers even when phrased differently

What this does not yet prove

  • It does not replace validation on a redacted real export
  • It does not prove every local model will perform the same way
  • It does not yet cover every free-form question staff might invent
  • It should still be treated as a controlled pilot capability, not a final production claim

Working with a coding agent

Workflow

A teacher and a coding agent can iterate on tools quickly, with the human setting the constraints

This study was built as a working conversation. The human set the questions, constraints, and quality bar. The coding agent handled the implementation and repeated optimisation work.

Teacher input

Define the real questions, the safety constraints, and what counts as a useful answer.

Agent work

Write code, generate synthetic linked data, run benchmarks, redesign tools, and package the outputs.

Shared loop

Review results together, inspect failures, and keep refining until the local setup is reliable enough to test further.

What this shows: a coding agent is most useful when staff bring the domain judgement and constraints, and let the agent handle the repetitive engineering work quickly.

Training path

Training

Staff could learn this workflow, but the training boundary should be synthetic data only

Capability building

  • Teachers or staff can be trained to use Codex or Claude Code as a coding partner for building and refining local tools.
  • The aim is not to make every teacher a developer, but to let a small group learn how to shape prompts, tools, and tests with AI support.
  • That creates a repeatable internal workflow for prototyping and improving school-specific tooling.

Data boundary

  • If a cloud coding model is used in training, it should only see synthetic data.
  • That synthetic data can be generated locally by the model, or produced through scripts and spreadsheet workflows such as Excel.
  • The purpose is to learn the workflow without touching school PII or operational records.
Practical boundary: use synthetic data first to train staff and refine tools, then move only to approved redacted school extracts once the workflow and guardrails are understood.

What this means for the school

Recommendation

Suitable for a controlled pilot, not a broad rollout

Step 1

Show the working shape

Use the synthetic environment and the current browser chat to show the question types working locally end-to-end.

Step 2

Validate on a safe real extract

Use a redacted or approved export from the school to check that the same tool design holds up with the real column names and data shape.

Step 3

Set the pilot boundary

Start with a controlled list of supported questions, defined users, and clear wording around what the assistant should and should not be used for.

Recommended message to stakeholders: “We now have evidence that the local approach can answer the current question set. The next step is a limited pilot on approved school data, alongside a safe training workflow that uses mock data only when staff are learning to build or refine tools.”

Can the school try this themselves?

Self-Testing

Two practical ways to test this in an LM Studio setup

Option 1 · easiest

Run the provided local chat

  • Install LM Studio and load the approved local model
  • Start LM Studio’s local OpenAI-compatible server
  • Run bridge.py against the sample data or a safe school export
  • Open the local browser chat and ask the same question set yourselves
Option 2 · more technical

Attach the tools into your own app

  • Run csv_tool.py --serve as a local HTTP tool service
  • Point your own app at LM Studio’s local API
  • Reuse the tool definitions from csv_tool.py
  • Let your app route tool calls locally, without sending data to a remote service
Important framing for the school: the simplest first test is not to plug this straight into a wider stack, but to run the local bridge against a safe extract and check whether it answers the school’s question set in the school’s own environment.
A step-by-step version of this is included in school_self_test_guide.md. The full project files are in the GitHub repository.
